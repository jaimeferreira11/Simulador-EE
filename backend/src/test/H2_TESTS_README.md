# H2 in-memory test profile

Optional test setup for running a small subset of integration tests **without Docker**.
Lives alongside the main Testcontainers-based `IntegrationTestBase`; does NOT replace it.

## When to use which base

| Need | Base |
|---|---|
| Auth / login / JWT flow (depends on BCrypt seed user) | `IntegrationTestBase` (Testcontainers) |
| Engine / Golden File / formulas | `IntegrationTestBase` |
| jsonb columns (`bot_config`, `notificacion.datos_extra`, audit `estado_anterior/nuevo`) | `IntegrationTestBase` |
| `pgcrypto` functions (`gen_random_bytes`) | `IntegrationTestBase` |
| Any controller that requires `/auth` to work | `IntegrationTestBase` |
| Pure repository CRUD on a simple table | `H2IntegrationTestBase` |
| Testing a service whose dependencies are easily mocked or scope-trimmed | `H2IntegrationTestBase` |

## Why H2 cannot replace Postgres for most tests

The MVP migrations are written for Postgres and use:

- `JSONB` columns with `jsonb_build_object`, `||` jsonb concat, `->`/`->>` operators
- `CREATE EXTENSION pgcrypto` + `gen_random_bytes`
- ~1500+ lines of `DO $$ ... $$ LANGUAGE plpgsql` seed scripts that build the dev/demo data set, including the BCrypt-hashed user `moderador@simulador.py / password123` consumed by every controller integration test
- Schema-qualified casts (`::TEXT`, `::jsonb`) and Postgres-only `INTERVAL` arithmetic
- `BIGSERIAL` (H2 needs `GENERATED BY DEFAULT AS IDENTITY` in PG mode)

H2's Postgres compatibility mode handles the basics (schemas, types, `BOOLEAN`, `TIMESTAMP WITH TIME ZONE`) but not the items above. Trying to make all migrations H2-compatible would require maintaining a parallel set, plus replacing `JdbcConversions` (which constructs `org.postgresql.util.PGobject` directly).

The pragmatic compromise: skip Flyway entirely for the H2 profile and load a minimal hand-written schema that covers only the tables a given test needs.

## How it's wired

| File | Purpose |
|---|---|
| `pom.xml` | Adds `com.h2database:h2` (test scope) |
| `src/main/java/py/simulador/config/JdbcConversions.java` | Annotated `@Profile("!test-h2")` so it's skipped under H2 |
| `src/main/java/py/simulador/config/H2JdbcConversions.java` | H2 variant: `JsonbValue` <-> `String` instead of `PGobject`. Active only under `test-h2` |
| `src/test/resources/application-test-h2.yml` | H2 datasource, Flyway disabled, points to `h2/schema.sql` + `h2/data.sql` |
| `src/test/resources/h2/schema.sql` | Hand-written minimal schema (currently: only `sim.entidad`) |
| `src/test/resources/h2/data.sql` | Tiny seed for ported tests |
| `src/test/java/py/simulador/H2IntegrationTestBase.java` | Abstract base. Excludes web/security autoconfig. Trimmed `@ComponentScan` (only `common` + `entidad`) |
| `src/test/java/py/simulador/entidad/EntidadRepositoryH2Test.java` | Proof-of-concept ported test |

## Running

```bash
cd backend

# Run only the H2 proof-of-concept test
./mvnw test -Dtest=EntidadRepositoryH2Test

# Run all unit tests (none require Docker; Surefire excludes *IntegrationTest/*E2ETest)
./mvnw test

# Run integration + E2E tests (requires Docker — Testcontainers + Postgres)
./mvnw verify
```

## Adding a new H2 test

1. Add the table(s) you need to `src/test/resources/h2/schema.sql`.
   - Translate Postgres-isms: `BIGSERIAL` -> `BIGINT GENERATED BY DEFAULT AS IDENTITY`; `JSONB` -> `CLOB` (use `JsonbValue` in entity).
   - Avoid `pgcrypto`, plpgsql, `INTERVAL`, jsonb operators.
2. If your table has FKs to tables that aren't in `schema.sql` yet, add those tables too (or drop the FK from the test schema).
3. Add any required seed rows to `src/test/resources/h2/data.sql`.
4. Add the package to `H2IntegrationTestBase.H2TestApplication.basePackages` if it's not already covered. Beware: scanning a new package may pull in beans that depend on websocket / security / mail wiring — you'll need to widen the `excludeFilters` regex or add `@MockBean`s.
5. Extend `H2IntegrationTestBase` from your test class.

## Known limitations

- The current `H2TestApplication` config bootstraps **all 32** Spring Data JDBC repositories (`Found 32 JDBC repository interfaces` in the log). They're created lazily and unused ones don't crash, but if a new test scans a package with a service that wires a repo whose table isn't in `schema.sql`, the bean will fail at first query.
- No security context: `H2IntegrationTestBase` excludes `SecurityAutoConfiguration`. To exercise auth-protected behaviour, use `IntegrationTestBase`.
- No web layer: `webEnvironment = NONE`. To exercise a controller, switch to a dedicated `@WebMvcTest` slice (a different pattern; not provided here).
